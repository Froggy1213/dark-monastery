## Architecture

**Dark Monastery** is a text-based Dark Fantasy RPG where the game master is Google Gemini. The player types actions (terminal or web UI); Gemini generates the next story beat as structured JSON, which the engine deserializes back into game state.

### Package map

| Package | Purpose |
|---|---|
| `internal/game` | `GameState` struct, `NewPlayer()` factory, `Engine`, `AIClient` interface, prompt builder, lore book |
| `internal/ai` | Gemini API client (`gemini-2.5-flash-lite`) and Embedding client (`gemini-embedding-001`) |
| `cmd/server` | Entry point (`main.go`), terminal game loop (`terminal.go`), web server startup (`web.go`) |
| `internal/handlers` | HTTP handlers (game actions, saves), WebSocket handler, session management |
| `internal/memory` | Dual-layer memory: short-term ring buffer (`History`) + long-term pgvector via `MemoryManager` + async `Extractor` worker pool |
| `internal/storage` | File-based saves (`FileStore`), in-memory cache (`MemoryStore`), PostgreSQL/pgvector (`PgStore`) |
| `static/` | Web frontend: HTML, CSS, JS (split into `js/` modules) |

### Data flow

```
Player input (Terminal/HTTP/WebSocket)
  → handlers or terminal loop
    → memory.MemoryManager.BuildContext()  — RAG context from short-term + long-term memory
    → ai.GeminiClient.SetMemoryContext()
    → game.Engine.ProcessTurn()
      → ai.GeminiClient.GenerateNextTurn()
        → game.BuildSystemPrompt()  — Lore + Memory
        → game.BuildUserPrompt()    — State JSON + Action
        → Gemini API (JSON mode)
        ← game.GameState
    ← updated *game.GameState
  → memory.MemoryManager.Add()  — save to short-term + async pgvector
  → storage.FileStore.SaveWithHistory()  — auto-save to disk
  → Response (JSON / WebSocket message / terminal print)
```

### Key design decisions

- **Gemini is the game engine.** There is no rule-based game logic — all story progression, stat changes, inventory updates, and location transitions are generated by the LLM.
- **State is a flat struct.** `GameState` has no nested objects or relationships — it's a simple bag of fields designed to be easy for the LLM to produce as valid JSON.
- **The system prompt is in Russian** and lives in `internal/game/prompt.go` (`BuildSystemPrompt`). It instructs Gemini to act as a strict game master, always return valid JSON matching the `GameState` schema, and drive the plot forward with atmospheric dark fantasy narration.
- **Exponential backoff** in `gemini.go` and `embeddings.go`: up to 3 retries on HTTP 429 (rate limit) and 503 (overload), with `attempt*2` second delays.
- **JSON mode** is enforced via Gemini's `responseMimeType: "application/json"` generation config.

### Memory system (RAG)

- **Short-term**: ring buffer of last 3 turns (`memory.History`), injected directly into prompt
- **Long-term**: PostgreSQL + pgvector with cosine similarity search. Each turn is asynchronously embedded by `memory.Extractor` worker pool (2 workers, queue of 100). Re-embedding of failed records retried every 30s.
- **Fallback**: If PostgreSQL is unavailable, the game runs in fallback mode with only short-term memory.

### Frontend structure

```
static/
  index.html        — Main page
  style.css         — Dark fantasy theme (design tokens, atmosphere, animations)
  js/
    effects.js      — Visual effects engine (particles, screen shake, etc.) → window.DarkFX
    state.js        — Central game state → window.GameState
    ui.js           — DOM rendering, story log, thinking indicator → window.GameUI
    websocket.js    — WebSocket connection + HTTP fallback → window.GameWS
    app.js          — Entry point, session management, event wiring
```

### Storage layers

1. **In-memory** (`storage.MemoryStore`) — cache for active sessions
2. **File-based** (`storage.FileStore`) — JSON files in `~/.dark-monastery/saves/`
3. **PostgreSQL+pgvector** (`storage.PgStore`) — persistent semantic memory

### Project phase

The project is in **Phase 2** (web-enabled). Terminal mode, web frontend, WebSocket API, file storage, and RAG memory are all functional. PostgreSQL/pgvector requires Docker (`docker-compose up -d`) or a running pgvector instance.
